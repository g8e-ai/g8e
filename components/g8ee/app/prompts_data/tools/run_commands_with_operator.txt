Execute shell commands on the connected system through the g8e Operator.

IMPORTANT: You DO NOT propose shell commands directly. The Tribunal — a five-member AI panel — is the sole authority for producing the command string. You describe what you want accomplished; the Tribunal translates that into precise shell syntax for the target OS/shell.

<parameters>
- request: REQUIRED — natural-language description of what the Operator must accomplish. Focus on investigative intent (what to learn, verify, or change). DO NOT write shell syntax here.
- guidelines: OPTIONAL — creative guidance for the Tribunal expressed in plain language: flags to favour, behaviours to avoid, edge cases to cover, tradeoffs you consider important. NEVER include shell syntax. Leave empty when no guidance is needed.
- target_operator: hostname, operator_id, or index ("0","1") — REQUIRED when multiple operators bound
- target_operators: run the Tribunal-generated command on multiple operators simultaneously (["all"] or ["host-1","host-2"] or ["0","1"])
- timeout_seconds: default 300 | expected_output_lines: for UI sizing
</parameters>

<examples>
GOOD request: "Show disk usage under /home broken down per-user, sorted largest first, top 10."
GOOD request with guidelines: request="Find all files modified in the last 24 hours under /var/log"; guidelines="Prefer a portable find invocation over locate; exclude compressed archives."
BAD (do not do this): request="du -sh /home/*/ | sort -rh | head -10"  — that is a command string, which is the Tribunal's job to write.
</examples>

<batch_execution>
target_operators=["all"] runs on all bound operators in parallel. Use for fleet-wide operations.
</batch_execution>

<data_sovereignty>
LFAA: Output is available in result fields after execution — already Sentinel-scrubbed. That scrubbed result is the authoritative view of the command output for analysis.
Call `fetch_execution_output` only when raw, unredacted output is required for analysis or the user explicitly requests it. Raw output is retained on the Operator's local storage for data sovereignty.
</data_sovereignty>

<never>
- Never call `fetch_execution_output` as a routine follow-up; the scrubbed result is the authorized default for analysis.
- Never treat Sentinel placeholders as missing data; they indicate presence of a value the user can confirm locally.
</never>

<sentinel_scrubbing>
All output is processed by g8e Sentinel before transmission. Sensitive data replaced with placeholders:
- Network: [IP_ADDR], [HOST], [URL], [MAC_ADDR]
- Credentials: [JWT], [AWS_KEY], [GITHUB_TOKEN], [SLACK_TOKEN], [OKTA_TOKEN], [CREDENTIAL], [BEARER_TOKEN]
- Cloud: [AWS_RESOURCE], [ACCOUNT_ID], [GCP_API_KEY], [AZURE_SECRET]
- PII: [EMAIL], [PHONE], [PII]
- Identifiers: [UUID], [PATH], [CONN_STRING]
- Data: [DATA_ROW], [JSON_DATA], [VALUE], [ENCODED_DATA]

Treat placeholders as indicators of data presence. Ask the user to provide specific values when needed. Structure hints, error types, and exit codes are preserved.
</sentinel_scrubbing>
