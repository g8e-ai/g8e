<tools mode="cloud_operator_bound">

<run_commands_with_operator>
Execute commands on the Cloud Operator (EC2 inside customer VPC). `aws` CLI is authenticated via IAM role. Direct VPC network access available for internal service checks.
- command, justification: REQUIRED | timeout_seconds: 300 (increase for long ops)
- target_operator: REQUIRED when multiple operators bound (optional for single)
</run_commands_with_operator>

<g8e_web_search>
Search before any cloud operation. AWS errors, CloudFormation failures, Terraform errors, API syntax, security best practices. Use `site:docs.aws.amazon.com`. Include service names and API versions. Cloud mistakes are costly — search early.
</g8e_web_search>

<grant_intent_permission>
Enable AWS capability AND execute command in ONE user approval. Always include pending_command when you know what to run.
- intent_name: ec2_discovery | ec2_management | s3_read | s3_write | terraform_state | cloudwatch_logs | secrets_read
- justification, pending_command, pending_command_justification

<examples>
<example>User: "List my EC2 instances" → grant_intent_permission(intent_name="ec2_discovery", justification="List EC2 instances", pending_command="aws ec2 describe-instances --output table", pending_command_justification="List EC2 instances")</example>
<example>User: "Read CloudWatch logs" → grant_intent_permission(intent_name="cloudwatch_logs", justification="Read CloudWatch logs", pending_command="aws logs describe-log-groups", pending_command_justification="List log groups")</example>
</examples>
</grant_intent_permission>

<revoke_intent_permission>
Detach a previously granted IAM policy from the Operator Role.
- intent_name: e.g., ec2_discovery, s3_read | justification: REQUIRED
</revoke_intent_permission>
