# g8e Scripts

The scripts layer provides the operational interface for the g8e platform. It abstracts the complexity of a multi-container environment into a unified CLI, ensuring consistent execution across development, testing, and production environments.

## Architecture Overview

g8e uses a **host-bridged execution model**. While the platform runs in Docker, most operational tasks are executed inside the `g8ep` (Platform) container to ensure environment consistency and access to internal APIs.

### The `./g8e` CLI Entry Point

The root `./g8e` script is a Bash-based dispatcher. It is the only script an operator should invoke directly on the host.

- **Dependency-Free Host:** The host requires only `docker` and `docker-compose`. No Python, Go, or Node.js toolchains are needed locally.
- **Auto-Bootstrapping:** If a command requires `g8ep` and it's not running, the CLI automatically builds and starts it.
- **Session Management:** Commands targeting the internal API (`data`, `security`, `mcp`) are gated by a local credential store (`~/.g8e/credentials`) which is populated via `./g8e login`.

### Execution Flow
1. **Host-Side:** Commands like `platform start` or `operator build` run directly on the host, orchestrating Docker Compose or triggering builds in specialized test-runners.
2. **Container-Side:** Commands like `data users list` or `security validate` are forwarded via `docker exec` into `g8ep`, where they have access to:
   - Internal service networks (`g8es`, `g8ed`).
   - Mounted secrets (TLS certs, internal auth tokens).
   - The full Python operational toolchain.

## Core Command Categories

### Platform Management (`./g8e platform`)
Orchestrates the lifecycle of the Docker Compose environment via `scripts/core/build.sh`.

- **`start` / `stop` / `restart`:** Basic container lifecycle. `start` waits for service health checks.
- **`rebuild`:** Recreates images using the layer cache. Automatically syncs agent personas from Python models to `agents.json` before rebuilding.
- **`reset`:** Destructive. Wipes all data volumes (g8es, g8ee, g8ed) and rebuilds from scratch. **TLS certificates in `g8es-ssl` are preserved.**
- **`wipe`:** Clears application data via the `g8es` API. Preserves platform settings, SSL certs, and authentication state. Useful for demo resets.
- **`logs`:** Aggregates logs from all managed services into a single, time-ordered stream via `scripts/core/logs.sh`.

### Data Management (`./g8e data`)
Unified interface for interacting with platform state, dispatched via `scripts/data/manage-g8es.py`.

- **Dispatcher Pattern:** The `manage-g8es.py` script routes requests to specialized modules (e.g., `manage-users.py`, `manage-store.py`).
- **`store`:** Direct CRUD on the `g8es` document store and KV engine.
- **`users` / `operators`:** Management of platform identities via the `g8ed` internal API.
- **`reputation`:** Manages the Tribunal's agent reputation state and Merkle-linked commitment chain.
- **`sync-personas`:** Unidirectional sync from Python persona models (`components/g8ee/app/models/personas/`) to the canonical `agents.json`.

### Security & Identity (`./g8e security`)
Manages the platform's root of trust and security invariants.

- **`validate`:** Checks TLS integrity, volume mount permissions, and environment variable consistency.
- **`certs`:** Manages the internal ECDSA P-384 CA. Certs are generated by `g8es` on first boot; this script handles rotation and trust installation.
- **`passkeys`:** Manages FIDO2/WebAuthn credentials for operator authentication.

### Testing (`./g8e test`)
Runs component-specific test suites in isolated runner containers.

- **Isolation:** Tests for `g8ee` (Python), `g8ed` (Node.js), and `g8eo` (Go) run in dedicated containers with the exact required toolchains.
- **Parity Check:** Every test run first triggers `scripts/testing/check_model_parity.py` to ensure Pydantic models match the JSON schemas used across components.

## Directory Structure

```text
scripts/
├── core/           # Platform lifecycle and logging (Bash)
│   ├── build.sh    #   Docker Compose orchestration
│   └── logs.sh     #   Log aggregation and filtering
├── data/           # Data operations (Python)
│   ├── _lib.py     #   Shared HTTP clients & auth logic
│   ├── manage-g8es.py # Main dispatcher
│   └── sync-personas.py # Persona model synchronization
├── security/       # TLS and Security (Bash/Python)
│   ├── manage-ssl.sh # Cert lifecycle
│   └── validate-platform-security.sh
├── testing/        # Test runners and parity checks
│   ├── run_tests.sh # Main test execution bridge
│   └── check_model_parity.py
└── tools/          # Interactive setup wizards
    ├── setup-llm.sh # LLM provider configuration
    └── setup-ssh.sh # SSH multiplexing for operator streaming
```

## Technical Invariants

### 1. Internal Authentication
All scripts running inside `g8ep` authenticate to `g8es` and `g8ed` using a shared `X-Internal-Auth` token. This token is mounted at `/g8es/internal_auth_token` and is automatically rotated via `./g8e security rotate-internal-token`.

### 2. Path Resolution
All scripts must resolve `PROJECT_ROOT` relative to their own location to ensure they work regardless of the user's current working directory.
- **Bash:** `$(cd "$(dirname "${BASH_SOURCE[0]}")/../.." && pwd)`
- **Python:** `Path(__file__).parent.parent.parent.absolute()`

### 3. Service Readiness
The platform does not consider itself "up" until Docker health checks return `healthy`. `build.sh` uses a background log-tailing pattern during `up` to provide visibility into why a service might be failing to reach readiness.
