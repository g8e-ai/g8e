# g8e Pub/Sub Architecture

Official wire-protocol channel reference for the g8e platform. Inter-component communication for command execution, status monitoring, and session bootstrap relies on these channels.

---

## Core Lifecycle

The pub/sub system facilitates the bidirectional communication between the central platform (`g8ee`) and distributed agents (`g8eo` Operators).

1.  **Bootstrap**: An Operator identifies itself and receives its configuration (either via HTTP or the `auth` channels).
2.  **Command Execution**: `g8ee` publishes a command to the Operator's specific `cmd` channel.
3.  **Result Delivery**: The Operator executes the command and publishes results back to the `results` channel.
4.  **Health Monitoring**: The Operator publishes periodic signals to the `heartbeat` channel to maintain its `AVAILABLE` status.

---

## Channel Formats

### Per-Operator-Session Channels

The canonical format for per-session communication is:
`{prefix}:{operator_id}:{operator_session_id}`

| Prefix | Source | Destination | Purpose |
| :--- | :--- | :--- | :--- |
| `cmd` | `g8ee` | `g8eo` | Delivery of executor commands and system requests. |
| `results` | `g8eo` | `g8ee` | Delivery of execution results, stdout/stderr, and errors. |
| `heartbeat` | `g8eo` | `g8ee` | Signal of life and current resource usage from the agent. |

**Invariants:**
- **Session ID verbatim**: The `operator_session_id` is the full string (e.g., `operator_session_1764000000_...`) generated by the platform.
- **Single separator**: Channels use `:` as a segment separator.
- **Bounded parsing**: Always use a bounded split (max 2 splits) when parsing to ensure session IDs containing `:` are not truncated.

### Session Auth Channels

Specialized channels used for agent bootstrap when HTTP is unavailable or for specific client flows.

- **Request**: `auth.publish:session:{session_hash}`
- **Response**: `auth.response:session:{session_hash}`

The `{session_hash}` is the SHA-256 hex digest of the `operator_session_id`.

---

## Platform Broadcast Channels

Used for global aggregation and real-time event streaming to frontends.

| Channel | Purpose |
| :--- | :--- |
| `operator_heartbeats` | Aggregated stream of all heartbeats for platform-wide monitoring. |
| `sse_events` | Server-Sent Events stream for browser frontend updates. |
| `system_events` | Critical system-level notifications (e.g., certificate revocation). |
| `g8eo_results` | *Deprecated*. Use per-session `results` channels instead. |

## Canonical Implementation

### Single Source of Truth
`shared/constants/channels.json` defines all prefixes and formats. All components MUST consume these values or their typed wrappers.

## Technical Safety Standards

1.  **No Hand-Rolling**: Never construct channel names using string interpolation or concatenation. Drift between components causes catastrophic delivery failure.
2.  **Refcounted Subscriptions**: `PubSubClient` (Python) implementers must use reference counting for subscriptions to prevent accidental early unsubscription when multiple services share a connection.
3.  **Mandatory TTL**: Inbound `auth` listeners must implement a TTL (default 300s) to prevent memory leaks from abandoned bootstrap attempts.
4.  **Correlation IDs**: Command-result parity is maintained via the `execution_id` inside the message payload, not just the channel identity.
