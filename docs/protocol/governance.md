# Governance & Mechanism Design

Last Updated: 2026-05-16
Version: v0.3.0

Agentic AI safety in g8e is framed as a **consensus problem**: given a population of LLM-instantiated personas with different lenses, a calibrated adversary among them, and a human user with finite attention, how do we converge on an executable command that is safe, audited, and minimally costly?

The mechanism operates through a strictly ordered L1/L2/L3 validation hierarchy where each layer provides a unique type of safety guarantee. On all protocol paths, this hierarchy is carried in the `GovernanceEnvelope` as metadata beside the typed payload.

The **g8e Protocol** is the substrate that defines these layers. Any conforming **Operator** implementation (the reference being `g8eo`) enforces them on the host. An optional **Application Layer** (the bundled Engine or any BYO consensus producer) supplies the proofs required to pass them.

## The L1/L2/L3 Governance Hierarchy

Every command generated by the system must pass through three distinct governance layers.

### L1: Technical Bedrock (Hard Gates)
L1 provides hardcoded, non-negotiable safety invariants that are enforced at the Operator boundary.

1.  **Protobuf Reflection (`forbidden_patterns`)**: 
    Enforced in `services/g8eo/internal/services/governance/transaction_verifier.go` via `validateL1Governance`. It scans incoming messages for the `forbidden_patterns` custom option defined in `protocol/proto/common.proto`. For example, the `command` field in `CommandRequested` prohibits `sudo`, `su`, and `rm -rf /`.
    
2.  **Sentinel Guard (Threat Analysis)**:
    Enforced in `services/g8eo/internal/services/sentinel/sentinel.go`. Sentinel performs pre-execution regex analysis against 90+ threat patterns mapped to the MITRE ATT&CK framework (e.g., reverse shells, privilege escalation, data exfiltration).
    
3.  **Fail-Closed Behavior**:
    If any L1 check fails, `g8eo` rejects the command before it reaches the shell, returning a `sentinel_blocked` or `validation_failed` error.

### L2: Consensus (Consensus)
The Consensus layer converts high-level intent into executable commands using an ensemble of independent agents (bundled in `g8ee` or provided by a BYO system).

1.  **Plurality Vote**: Candidates are aggregated and ranked. A winning candidate must achieve consensus.
2.  **Tribunal Signature**: 
    The Tribunal attaches an ED25519 signature from a trusted L2 signer over `transaction_hash|decision` (where decision is a boolean).
    `g8eo` verifies this signature in `services/g8eo/internal/services/governance/transaction_verifier.go`. It uses an Operator-owned `SignerStore` for public key lookup. The verifier rejects every missing-key or unknown-key transaction.

### L3: Authorization (Human Gate)
L3 involves human authorization, governed by the **Auditor-User Partition**.

*   **Proof of Human Presence (PHP)**: Mutations (Bash, File Edit, etc.) require a hardware-bound WebAuthn passkey assertion. The assertion challenge is the transaction's `transaction_hash`. The proof is carried in `governance.l3.proof`.
*   **Auto-Approval**: 
    Defined in `@/home/bob/g8e/services/g8ee/config/auto_approved.json`. Benign diagnostic base commands (e.g., `uptime`, `df`) can bypass the human prompt if `enable_auto_approve` is active. This is reflected as `l3.auto_approved = true`.
*   **Safety Invariant**: Auto-approval **NEVER** bypasses L1 or L2. It only represents L3 authorization state for commands already verified as technically safe and low-risk.

## The Warden & Execution Boundary

The **Warden** is the final execution gateway in `g8eo`. It enforces a strict fail-closed lifecycle for all mutations:

1.  **Pre-Execution Receipt**: Before dispatching the command, the Warden creates and signs an `ActionReceipt` with status `EXECUTION_STATUS_EXECUTING`. This receipt is logged to the `AuditVault`. If logging fails, execution is aborted.
2.  **Execution**: The command is dispatched to the `ExecutionHandler`.
3.  **Post-Execution Receipt**: Once finished, the Warden updates the receipt with the final status (`COMPLETED` or `FAILED`), captures the `state_root_after`, and signs it again.
4.  **Audit Integrity**: The dual-receipt model ensures that every attempt to mutate reality is cryptographically recorded, even if the process crashes mid-execution.

## Protocol Binding

The governance hierarchy is bound to the command protocol via `@/home/bob/g8e/protocol/proto/common.proto`:

| Governance layer | Protocol representation | Enforcement point |
|---|---|---|
| L1 Technical Bedrock | `governance.l1`, `forbidden_patterns`, `expires_at`, `nonce` | `g8eo` reflection, expiry, and replay check |
| L2 Consensus | `governance.l2.tribunal_signature` | `g8eo` signature verification |
| L3 Authorization | `governance.l3.proof` | `g8eo` WebAuthn L3 enforcement for mutations |
| State Freshness | `state_merkle_root` | `g8eo` comparison to local ledger root |

## Wire Format and Signing Invariant

The canonical wire format for all client-facing surfaces is **canonical JSON (protojson)** for `GovernanceEnvelope`. 

**Signing basis**: Deterministic `transaction_hash` computed from normalized envelope fields (excluding the hash and id fields themselves). The verifier enforces `id == transaction_hash`.

**Canonicalization rules** (implemented in `@/home/bob/g8e/services/g8eo/pkg/uap/types.go`):
- Field names in proto definition order
- Strings as UTF-8
- Numbers as decimal integers
- Bytes as base64
- Result hashed with SHA-256

## Core Safety Invariants

1.  **Fail Closed**: Any missing proof, expired transaction, or failed audit write results in immediate rejection.
2.  **Auditor-User Partition**: The machine handles correctness; the human handles intent. Neither is sufficient alone.
3.  **ActionReceipt Integrity**: Every mutation emits a signed receipt that cryptographically binds the outcome to the original intent and the system state.
